# Frontend (Calendar Booking App) — Implementation Plan

## Context

The repo has an OpenAPI contract at [spec/openapi.yaml](spec/openapi.yaml) (generated from [../typespec/main.tsp](../typespec/main.tsp)) describing a small Calendly-style scheduling service. The product brief in [spec/task.txt](spec/task.txt) defines two roles with no auth:

- **Owner** — predefined profile; manages event types and views all upcoming bookings.
- **Guest** — picks an event type, sees available slots in the next 14 days, books a slot. No two bookings can share the same time.

The `frontend/` folder is empty apart from the spec. Goal: scaffold a **Vite + React + TypeScript + Mantine** SPA against this OpenAPI contract, with **Prism** mocking the backend so the full UX is exercisable today.

## Stack

| Concern | Choice |
|---|---|
| Build tool | Vite |
| Language | TypeScript (strict) |

| Routing | React Router v7 (library / declarative mode — package name is `react-router`, not `react-router-dom`) |
| Server state | TanStack Query v5 |

| Dates / TZ | `dayjs` with `utc` and `timezone` plugins (Mantine dates already requires dayjs) |
| Dev mock server | [Prism](https://github.com/stoplightio/prism) (`@stoplight/prism-cli`) running on port 4010, proxied by Vite |
| Lint/format | ESLint + Prettier (Vite React-TS template defaults) |

## Routing

Single `AppShell` with header switching between Guest (default) and Owner sections.

| Path | Component | Role | API |
|---|---|---|---|
| `/` | `EventTypesPage` | Guest | `GET /calendar/event_types` |
| `/book/:eventTypeId` | `BookEventPage` | Guest | `GET /calendar/event_types/{id}/available_slots`, `POST /calendar/scheduled_events` |
| `/book/:eventTypeId/confirmed` | `BookingConfirmedPage` | Guest | — |
| `/admin/event-types` | `EventTypesAdminPage` | Owner | `GET` + `POST /calendar/event_types` |
| `/admin/bookings` | `ScheduledEventsPage` | Owner | `GET /calendar/scheduled_events` |

## API integration

1. `npm run gen:api` runs `openapi-typescript spec/openapi.yaml -o src/api/schema.ts`.
2. `src/api/client.ts` exports a singleton:
   ```ts
   import createClient from "openapi-fetch";
   import type { paths } from "./schema";
   export const api = createClient<paths>({ baseUrl: "/api" });
   ```
3. `src/api/hooks.ts` wraps each operation in TanStack Query (`useEventTypes`, `useAvailableSlots`, `useCreateEventType`, `useScheduledEvents`, `useCreateScheduledEvent`). Mutations invalidate the corresponding lists.
4. `vite.config.ts` proxies `/api` → `http://127.0.0.1:4010` so Prism receives the calls verbatim.

## Mocking via Prism

Add as devDependency. Two npm scripts:

- `mock`: `prism mock spec/openapi.yaml -p 4010 --dynamic` — generates random but schema-valid responses.
- `dev`: `concurrently -k "npm:mock" "vite"` — single command to bring up the whole dev stack.

Prism is stateless, so creating a booking and then listing bookings won't show the new entry. That's acceptable for UI development and matches the contract's shape; a real backend will follow.

## Domain logic on the client

- **Timezone**: read once via `Intl.DateTimeFormat().resolvedOptions().timeZone` (already noted in [../notes.txt](../notes.txt)). Pass as `clientTimeZone` query param when fetching available slots; submit as `guestTimezone` on booking creation.
- **14-day window**: filter/clamp `slotsPerDay` to `[today, today + 13 days]` defensively, even though the server is the source of truth.
- **Slot collision**: not enforced client-side — the server owns the rule. The mutation surfaces 4xx as a Mantine notification; on success we navigate to the confirmation page.
- **Owner identity**: no auth, no profile screen — the brief says the owner is predefined. The Owner section is just two pages behind a header link.

## UI sketches

- **Guest `/`**: grid of `EventTypeCard`s (title, duration badge, description, "Book" button).
- **Guest `/book/:id`**: two-pane layout — left `DatePicker` (Mantine) limited to the 14-day window with non-available days disabled; right column lists time slots for the selected day. Selecting a slot opens a Mantine `Modal` with a name + email form, then submits.
- **Owner `/admin/event-types`**: table of existing event types + a "New event type" `Modal` form (`EventTypeForm` using `@mantine/form`, fields: name, description, durationMinutes).
- **Owner `/admin/bookings`**: sortable table grouped by date showing event type, guest name/email, start time in owner's local TZ, duration.

## Implementation order

1. **Scaffold**: `npm create vite@latest frontend -- --template react-ts`, then install runtime deps (`@mantine/core @mantine/hooks @mantine/dates @mantine/form @mantine/notifications dayjs react-router @tanstack/react-query openapi-fetch`) and dev deps (`postcss postcss-preset-mantine openapi-typescript @stoplight/prism-cli concurrently`). Add `postcss.config.cjs` with `postcss-preset-mantine` + `postcss-simple-vars` per Mantine v9 docs, and put `<ColorSchemeScript />` in `index.html`.
2. **Wire providers** in [src/main.tsx](src/main.tsx) (`MantineProvider` → `Notifications` → `QueryClientProvider` → `BrowserRouter`) with the required Mantine CSS imports, and create the AppShell in [src/App.tsx](src/App.tsx).
3. **Generate API types** and write [src/api/client.ts](src/api/client.ts) + [src/api/hooks.ts](src/api/hooks.ts).
4. **Set up Prism + Vite proxy** and the `dev` script.
5. **Guest flow**: EventTypesPage → BookEventPage → BookingConfirmedPage.
6. **Owner flow**: EventTypesAdminPage, ScheduledEventsPage.
7. **Polish**: loading/error states via Mantine `Loader` + `Notifications`, empty-state messages, basic responsive checks.

## Verification

End-to-end, all manual (no backend yet, so no automated integration tests in this iteration):

1. `cd frontend && npm install`
2. `npm run gen:api` — confirm `src/api/schema.ts` is generated with no errors.
3. `npm run dev` — Prism starts on 4010, Vite on 5173.
4. Open `http://localhost:5173/`: event types load (Prism returns mock list).
5. Click an event type → 14-day window renders, picking a slot opens the booking modal, submission shows a success toast and routes to `/book/:id/confirmed`.
6. Visit `/admin/event-types`: list loads, "New event type" creates one (Prism echoes it back; list won't grow — expected).
7. Visit `/admin/bookings`: list loads.
8. `npm run build && npm run preview` — production build succeeds and the preview server renders the same flows.

Once a real backend is available, swap the Vite proxy target; no app code should need to change.
